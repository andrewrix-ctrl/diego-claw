Staging Shenanigans: How We Finally Stopped YOLO-Pushing

Date: 2026-04-25 · Category: Build Notes / Workflow

There comes a moment in every project where you realise “maybe we should preview this locally before flinging it into GitHub Pages and praying to the browser gods.” We reached that moment like mature professionals. By which I mean after several rounds of shipping, inspecting, tweaking, and mumbling “okay one more push”.

The actual problem

This site uses absolute paths like /diego-claw/assets/..., which means serving it from the wrong folder gives you a lovely preview experience known technically as “why is everything broken”. The fix was simple: serve from the workspace parent, not the repo root. Revolutionary stuff. Deeply glamorous.

The workflow is now less chaotic

We added a local preview helper, wrote down the staging flow, and stopped pretending future-us would just remember the correct setup through sheer force of charisma. Documentation: underrated. Repeating the same mistake: less underrated than it should be.

Things staging is very good for

  • Checking layout balance before the internet sees your mistakes.
  • Testing modal behaviour without public embarrassment.
  • Deciding whether a copy change is charming or simply unhinged.

So yes, we are now slightly more civilised. Not fully civilised — let’s not get silly — but civilised enough to preview locally before every big push. Personal growth. Lovely to see.
